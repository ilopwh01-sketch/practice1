# Ralph Loop 정리

Ralph Loop는 AI 코딩 에이전트를 한 번 실행하고 끝내는 방식이 아니라, 같은 목표를 여러 번 반복 실행하면서 작업을 점진적으로 완성하는 에이전트 루프 기법입니다.

핵심 아이디어는 단순합니다.

```bash
while not_done; do
  ai_agent "$PROMPT"
done
```

매 반복마다 프롬프트의 큰 목표는 유지되지만, 코드베이스와 작업 기록은 계속 변합니다. 그래서 에이전트는 매번 새 컨텍스트로 시작하면서도 Git 기록, 작업 계획, 테스트 결과, 파일 상태를 읽고 이전 작업 위에서 계속 전진할 수 있습니다.

## 왜 Ralph Loop라고 부르나

Ralph Loop는 Ralph Wiggum technique이라고도 불립니다. 이름은 단순하고 반복적인 방식이지만 의외로 효과적인 작업 패턴을 강조하기 위해 붙은 것으로 설명됩니다.

이 기법의 철학은 AI 에이전트의 한계를 억지로 숨기지 않는 것입니다. 긴 대화가 누적되면 컨텍스트가 지저분해지고, 이전 오류와 수정 내역이 섞여 품질이 떨어질 수 있습니다. Ralph Loop는 세션을 반복적으로 새로 시작해 이 문제를 줄입니다.

## 작동 방식

1. 에이전트가 작업 지시서나 프롬프트를 읽습니다.
2. 현재 코드, Git 기록, 작업 계획, 테스트 결과를 확인합니다.
3. 가장 중요한 다음 작업을 하나 고릅니다.
4. 구현, 수정, 테스트, 문서화를 수행합니다.
5. 결과를 파일이나 Git 커밋으로 남깁니다.
6. 세션이 종료됩니다.
7. 완료 조건을 만족하지 못하면 새 세션으로 다시 시작합니다.

중요한 점은 상태가 대화창 안에만 있지 않다는 것입니다. 상태는 파일, 계획서, 테스트 결과, Git 기록처럼 다시 읽을 수 있는 외부 기록으로 남아야 합니다.

## Human-in-the-loop와의 차이

Human-in-the-loop는 사람이 매 단계 판단에 들어가는 구조입니다.

Ralph Loop는 에이전트가 스스로 반복 실행하면서 진전하고, 사람은 중간중간 방향을 조정하거나 최종 검토를 하는 구조에 더 가깝습니다. 그래서 더 정확히는 human-on-the-loop에 가까운 운영 방식으로 볼 수 있습니다.

다만 품질/보안/업무 데이터가 중요한 프로젝트에서는 사람이 완전히 빠지면 안 됩니다. Ralph Loop를 쓰더라도 승인 기준, 중단 조건, 리뷰 단계는 별도로 둬야 합니다.

## 장점

- 긴 세션의 컨텍스트 오염을 줄일 수 있습니다.
- 반복 작업을 에이전트에게 오래 맡길 수 있습니다.
- 테스트 실패, TODO, 작업 계획을 다음 반복에서 다시 읽고 수정할 수 있습니다.
- 작업 결과가 Git 기록으로 남아 추적하기 쉽습니다.
- 명확한 성공 기준이 있는 개발/분석 자동화에 잘 맞습니다.

## 위험과 주의점

- 목표와 종료 조건이 흐리면 무한 반복하거나 엉뚱한 방향으로 갈 수 있습니다.
- 테스트나 검증이 약하면 잘못된 결과를 계속 쌓을 수 있습니다.
- 너무 큰 작업을 한 번에 맡기면 반복해도 품질이 안정되지 않습니다.
- 민감 데이터가 있는 업무 프로젝트에서는 저장소 공개 범위, 토큰 권한, 로그 노출을 조심해야 합니다.
- 사람이 리뷰하지 않은 변경을 바로 운영 반영하면 위험합니다.

## 한화큐셀 업무자동화 실습에 적용한다면

이 저장소에서는 Ralph Loop를 바로 무인 실행하기보다, 다음 정도의 구조로 실습하는 것이 안전합니다.

1. `TASKS.md`에 작은 작업 단위 목록을 만든다.
2. 각 작업은 하나의 분석 산출물이나 README 개선처럼 작게 쪼갠다.
3. 에이전트는 한 번에 한 작업만 처리한다.
4. 처리 후 `git diff`, 리포트 수치, HTML 출력 여부를 확인한다.
5. 사람이 결과를 검토한 뒤 다음 작업으로 넘어간다.

예시 작업:

- `analysis_report.md`의 핵심 결론을 README에 요약
- `analysis_summary.html` 설명 추가
- 불량코드 파레토 표를 문서화
- 설비별 NG율 상위 항목을 액션 아이템으로 변환
- 결과 검토 체크리스트 작성

## 추천 운영 규칙

- 반복 횟수 제한을 둔다.
- 완료 조건을 문장으로 명확히 쓴다.
- 매 반복마다 테스트나 검증 명령을 실행한다.
- Git 커밋을 작게 나눈다.
- 민감 정보는 커밋하지 않는다.
- 사람이 최종 리뷰한 결과만 공유한다.
